# Особенности архитектуры
## Максимальная инкапсуляция и модульность
Каждый логический элекмент проекта должен быть максимально изолирован от других и легко заменён на аналогичный. 

Пример 1
Если сейчас основной интерфейс - это телеграм-бот, то завтра это может быть веб-приложением на streamlit, а послезавтра - что-то совершенно новое. 
А для отладки можно вообще консоль использовать. 
Или всё это использовать одновременно и переключать через параметр

Пример 2
Базовая опция логирования - postgres. Но завтра может потребовать перейти на slq-lite (для компактности) или clickhouse (для масштабирования). 
Нужно хранить все параметры и изолировать функции так, чтобы не завязываться на особенности функционала конкретных баз данных или даже отколючить логирование целиком.

Пример 3
Для работы могут использовать сторонние API. Эти провайдеры тоже могут меняться. Процесс получения и обработки данных оттуда и сама внутреняя логика должны быть надёжно разнесены

## Документируем всё
Пишем максимально подробные комментарии на русском языке. У каждого компонента пишем его цель, назначение, основную мысль и то, что получается на выходе.
Если нужно выполнить какие-то необычные вещи, то указываем это в комментарии явно.
Если тут есть сложность, опасность  или риски необычного поведения в будущем - указываем это явно в комментариях.
Если есть что-то, что можно будет улучшить в будущем, но сейчас нет в промте, то пишем TODO комментарии (и отдельно выносим их в будущий таск-лист).

## Гибкое отношение к зависимостям
Над проектом не будет постоянной поддержки, поэтому улучшения, проверки и дополнения - всегда проблема. 
Если есть возможность взять внешний функционал, который проверен, который постоянно развивается, разработчики которого могут учесть больше мелочей - нужно использовать.
Например, для создания ботов в телеграм лушче использовать aiogram, а не просто API telegram. 

Но, разумеется, не имеет смысла запаковывать в проект большую внешнюю библиотеку типа sklearm ради одногой простой функции типа max()

## Производительность и асинхронность
Пытаемся находить оптимальное решение по памяти и нагрузке процессеора среди возможных. Но с сохраненияем условий выше. 
Не обязательно найти иделаьный оптимальный алгоритм. Важно не остановиться на откровенно плохом и неудачном. 

Сразу держим в фокусе асинхронность вызовов. Ожидаем, что продуктом одновременно будут пользоваться тысячи пользователей.

# Внешний вид
Приложение, спрашивает как вас зовут, запоминает это, после этого отправляет в главное меню. В телеграм - предлагает по умолчанию user_name из телеграм.

В главном меню 4 пункта: 
- заполнить основную анкету
- заполнить дополнительную анкету
- найти страну, близкую вам по ценностям
- понять своё место в социуме

Заполнение основной анкеты - последовательные ответы на вопросы из анкеты /home/roman/python/wvs_bot/questions.json секция main_questions. Можно прерваться в любой момент, позже вернуться, продолжить с того же места, где и начали.
Ответы пишутся в специальную таблицу user_answers. Когда анкета заполнена, становятся доступны 3 и 4 пункт меню, до этого там лишь сообщения-"затычки".

Заполнение дополнительной анкеты (там же, секция secondary_questions) - последовательные ответы которые более тесно связаны с пользователем. 


Логирование в postgres, выключается по параметру в yaml.
Вывод по умолчанию в streamlit, может измениться на telegram или консоль по параметру в yaml.

В таблице gen_sample - данные из последней волны опроса социалогов, приведённые к виду, в котором можно сравнивать с ответами пользователей. 

event_name для логирования:
- start_screen_visit — экран представления (новый пользователь)
- registration — user_name записан в базу
- main_menu_visit — пользователь на главном меню (4 пункта), без параметров
- main_menu_click — клик «В главное меню», параметр screen (экран-источник)
- main_questionary_start — первый экран основной анкеты, без параметров
- secondary_questionary_start — первый экран доп. анкеты, без параметров
- question_show — показ вопроса: questionary, qv_number, qv_id
- answer_sent — ответ записан: questionary, qv_number, qv_id, qv_text, answer
- find_counry_start — экран «Найти страну»; параметр answer (текст ответа пользователю)
- find_own_place_start — экран «Понять своё место в социуме»; параметр answer
- country_plot_loaded — карточка страны после графика; тайминги sql_ms, processing_ms, render_ms, country_plot_loaded_ms, total_ms


Тестирование логики:
- все кнопки кликабельны
- все нужные действия логируются
- параметры в users перезаписываются
- нет коллизий по user_id

- лаг в телеграм меньше 8 секунл
